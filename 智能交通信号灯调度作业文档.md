# 智能交通信号灯调度作业文档

> 本文档根据当前目录下 33 张课程作业网页截图按时间顺序提取、整理与总结生成。原始截图文件名形如 `截屏2026-05-30 11.xx.xx.png`，内容来自腾讯开悟“智能交通信号灯调度 开发指南”。

## 1. 作业概览

本作业要求开发一个强化学习智能体，用于控制单交叉口交通信号灯的相位与持续时间，使车辆在不同复杂度的交通环境中更高效地通过路口。

核心目标是：

- 根据环境返回的交通状态观测，判断当前路口的车辆、车道、信号灯和拥堵情况。
- 输出合法的信号灯控制动作，包括相位选择和持续时间。
- 通过训练优化策略，使平均车辆延误、排队长度、等待时间和信号切换惩罚等指标更优。
- 使用腾讯开悟强化学习框架完成环境交互、特征处理、奖励设计、算法训练、模型保存与评估。

本作业本质上是一个单路口交通信号灯控制问题。智能体需要在保证动作合法的前提下，尽可能减少交通参与者在路口处的延误和排队。

## 2. 场景与地图

场景为一个城市十字交叉口，包含车辆、进口/出口车道、交叉路口和信号灯。智能体在地图中拥有全局视角，可以获取当前环境对外开放的观测信息。

主要元素如下：

| 元素 | 说明 |
| --- | --- |
| 车辆 | 任务开始后，各类型车辆从距离路口较远的生成点出发，通过该交叉口，最终驶向目标点。 |
| 进口/出口车道 | 车辆从进口车道进入路口，经过交叉口后进入出口车道。 |
| 交叉路口 | 车辆汇集、转向和疏散的核心区域，信号灯控制会直接影响车流通行效率。 |
| 信号灯 | 智能体控制的实体，通过改变相位和持续时间疏解车流。 |

## 3. 计分规则

任务会在训练或评估结束后，根据交通运行表现计算得分。截图中给出的核心指标包括车辆延误、车辆排队长度、车辆等待时间和信号变换频率。

### 3.1 结束条件

- 正常结束：智能体预测步数达到用户设定的最大步数。
- 超时结束：场景等待中的车辆达到设定的最大等待车辆数，同时持续时间达到设定的最大等待车辆持续时长。

### 3.2 车辆延误

车辆延误用于描述车辆受交通设施和交通流环境影响后，相比自由流状态损失的时间。统计口径为车辆在交叉口进口车道上的平均延误。

可理解为：

```text
delay = actual_travel_time - free_flow_travel_time
free_flow_travel_time = travel_distance / max_speed
```

得分中会使用平均延误的倒数形式，延误越低，得分越高。

### 3.3 排队长度

排队长度统计交叉口进口车道上处于等待状态的车辆数量。截图中等待状态判断与低速阈值相关，车辆速度小于等于约 `0.1 m/s` 时会被视为排队或等待车辆。

平均排队长度越小，控制策略越优。

### 3.4 等待时间

等待时间统计进口车道上所有车辆的平均等待时间。与排队长度相比，它更关注车辆处于等待状态的累计持续时间。

平均等待时间越短，说明信号灯调度对车流疏解越有效。

### 3.5 信号变换频率

信号变换频率统计连续两个绿灯相位之间切换间隔小于 8 秒的次数。频繁切换会带来惩罚，因为这通常会导致交通流不稳定，也不符合真实信号控制习惯。

### 3.6 总分结构

截图给出的总分形式可概括为：

```text
总得分 =
  (车辆平均延误得分
 + 车辆平均排队长度得分
 + 车辆平均等待时间得分) * 1000
 - 信号变换频率得分
```

其中延误、排队和等待时间一般按 `1 / (1 + 指标 / 权重)` 转换为得分；信号变换频率则按频率乘权重扣分。

## 9. 时间关系

截图中说明了 `step`、`frame` 与仿真时间的关系：

- `frame` 是场景完整更新周期的时间单位。
- `step` 是强化学习环境中的一次动作交互单位。
- 本环境中 `1 step = 10 frame`。
- `1 frame` 约等于 `66-68 ms`。

因此，智能体并不是每一帧都重新决策，而是每个 step 输出一次动作，由环境在后续多个 frame 中执行并更新交通状态。

## 12. 强化学习训练流程

训练流程可以概括为四个阶段：

1. 智能体与环境循环交互：环境提供观测和奖励，智能体输出动作，环境执行动作并返回下一状态。
2. 样本处理：每一局 episode 生成一条轨迹，轨迹被转换为训练样本 `SampleData`。
3. 模型迭代优化：算法根据训练样本计算 loss 并更新模型参数。
4. 智能体模型更新：智能体加载或同步最新模型，继续与环境交互或参与评估。

核心交互逻辑如下：

```python
obs, state = env.reset(usr_conf=usr_conf)
act_data = agent.predict(list_obs_data=[obs_data])[0]
act = agent.action_process(act_data)
frame_no, _obs, score, terminated, truncated, _state = env.step(act)
```

训练中会将每一步构造成样本：

```python
frame = Frame(
    obs=obs_data.feature,
    _obs=_obs_data.feature,
    act=act,
    rew=reward,
    done=done,
)
```

当一局 episode 结束后，轨迹会经过 `sample_process()` 转换为训练样本，再输入 `agent.learn()`。

截图中的参考实现思路是：

- 使用网格化方法编码路口附近车辆位置。
- 使用车辆速度除以最大速度进行归一化。
- 将位置特征和速度特征拼接为最终 feature。

位置特征会根据车道方向与车辆行驶方向，将车道切分为若干网格，并把车辆映射到对应网格中。速度特征采用类似网格结构，但填入的是归一化后的速度。

## 14. 算法开发

开发目录：

```text
<智能体名称>/algorithm/algorithm.py
```

核心函数为 `learn()`，输入为训练样本列表：

```python
def learn(self, list_sample_data):
    loss = 0
    loss.backward()
    self.optimizer.step()
```

截图中提到框架提供了三个核心算法参考方向：

| 算法 | 特点 |
| --- | --- |
| DQN | 将 Q-Learning 与深度神经网络结合，适合离散动作空间，可使用经验回放提升稳定性。 |
| Target-DQN | 在 DQN 基础上增加目标网络和策略网络，提升训练稳定性。 |
| PPO | 策略梯度方法，通过限制策略更新幅度提高训练效率和稳定性。 |

监控指标中，DQN/Target-DQN 关注：

- `reward`：累计回报，应随训练逐步上升。
- `q_value`：目标网络 Q 值，用于观察 Q 值稳定性。
- `value_loss`：loss 指标，正常情况下应逐渐接近 0 或趋于稳定。

PPO 关注：

- `reward`：累计回报。
- `value_loss`：价值函数预测误差。
- `policy_loss`：策略函数优化目标。
- `entropy_loss`：策略探索性和多样性。

## 17. 智能体开发

开发目录：

```text
<智能体名称>/agent.py
```

智能体核心接口包括：

| 函数 | 作用 |
| --- | --- |
| `learn()` | 调用算法模块，用训练样本更新模型。 |
| `predict()` | 训练时使用，通常可带探索策略。 |
| `exploit()` | 评估时使用，选择最优动作或贪心动作。 |
| `load_model()` | 加载已保存模型。 |
| `save_model()` | 保存当前模型参数。 |

### 17.1 `learn()`

在单机训练中，开发者通常在训练工作流中手动调用 `learn()`；分布式训练中，训练函数可能会循环执行，但发送样本到样本池仍需要在工作流中完成。

### 17.2 `predict()`

`predict()` 接收 `ObsData` 列表，返回 `ActData` 列表。训练时可在该函数中加入随机探索，例如 epsilon-greedy 或基于概率分布采样。

### 17.3 `exploit()`

`exploit()` 用于评估，输入通常是原始环境观测字典，输出环境可直接执行的动作指令。该函数应尽量使用稳定的贪心策略，不应引入过多随机性。

### 17.4 `load_model()` 与 `save_model()`

模型加载和保存通常以 `model.ckpt-{id}.pkl` 为文件名：

```python
model_file_path = f"{path}/model.ckpt-{str(id)}.pkl"
```

保存时需要将模型状态拷贝到 CPU 后写入文件，以便在不同设备或平台中加载。

## 18. 模型保存限制与评估模式

截图中说明了模型保存限制：

- 保存模型频率限制：约 `2 次/分钟`。
- 单个任务保存模型次数限制：DQN、Target-DQN、PPO、DIY 均约 `200 次`。

评估模式中，可以在腾讯开悟平台创建模型评估任务，并配置环境参数。训练模式一般调用 `agent.predict()`，评估模式调用 `agent.exploit()`。因此，如果两者行为不同，需要分别实现并测试。

## 19. 分布式计算框架

截图中介绍的分布式框架为 KaiwuDRL，核心组件如下：

| 组件 | 功能 |
| --- | --- |
| `Environment` | 环境服务组件，负责运行强化学习环境并返回观测。 |
| `Aisrv` | 训练流程中枢，收集训练样本，运行训练和评估工作流，并协调数据传输。 |
| `Actor` | 预测服务组件，处理 Aisrv 的预测请求，调用智能体 `predict()` 或 `exploit()`。 |
| `Learner` | 训练服务组件，采集训练样本并调用 `learn()` 完成模型迭代。 |
| `MemoryPool` | 样本池，负责存储训练样本并向 Learner 提供样本。 |
| `ModelPool` | 模型池，负责存储模型文件并向 Actor 同步最新模型。 |
| 日志 | 采集系统运行日志。 |
| 监控 | 采集资源使用率和训练指标。 |

分布式预测服务流程：

1. `Aisrv` 向 `Environment` 发送环境配置并创建 episode。
2. `Environment` 返回原始观测。
3. `Aisrv` 基于原始观测向 `Actor` 发送预测请求。
4. `Actor` 调用智能体函数生成动作。
5. `Aisrv` 使用动作与 `Environment` 交互，环境返回新的观测。

分布式训练服务流程：

1. `Aisrv` 持续产生轨迹数据并完成样本处理。
2. 样本发送到 `MemoryPool`。
3. `Learner` 从样本池拉取样本训练。
4. `Learner` 将最新模型参数同步到 `ModelPool`。
5. `Actor` 周期性从模型池获取新模型。

## 20. 分布式训练配置

截图中的 `conf/configure_app.toml` 包含样本池和模型同步配置：

```toml
[app]
learner_train_sleep_seconds = 0.001
replay_buffer_capacity = 4096
preload_ratio = 1.0
reverb_remover = "reverb.selectors.Fifo"
reverb_sampler = "reverb.selectors.Uniform"
train_batch_size = 2048
dump_model_freq = 1000
model_file_sync_per_minutes = 1
modelpool_max_save_model_count = 1
```

关键含义：

- `replay_buffer_capacity`：样本池容量。
- `train_batch_size`：Learner 每次训练使用的样本批大小。
- `dump_model_freq`：模型保存频率。
- `model_file_sync_per_minutes`：Learner 与 Actor 之间的模型同步频率。
- `reverb_sampler`：从样本池采样的策略，可影响训练样本分布。

## 22. 策略类型选择

截图中对比了 On-Policy 与 Off-Policy：

| 特性 | On-Policy | Off-Policy |
| --- | --- | --- |
| 数据来源 | 使用当前正在训练的策略与环境交互产生的新数据。 | 可使用历史经验数据，数据可能来自旧版本策略或不同探索策略。 |
| 数据使用 | 一次性为主，策略更新后旧数据通常丢弃。 | 可重复利用，样本存储在经验回放缓冲区中。 |
| 主要优势 | 训练稳定，理论收敛性好，行为更易理解。 | 数据效率高，适合交互成本较高的场景。 |

配置示例：

```toml
algorithm_on_policy_or_off_policy = "on-policy"
dump_model_freq = 1
```

交通信号灯调度任务通常可以从 DQN/Target-DQN 这类 Off-Policy 方法入手，因为历史样本复用能降低交互样本成本；若使用 PPO，则应重点处理好策略更新频率和样本新鲜度。

## 26. 可采用的奖励设计示例

可将奖励设计为多个交通指标的加权组合：

```text
reward =
  - w_delay * avg_delay
  - w_queue * avg_queue_length
  - w_wait * avg_waiting_time
  - w_switch * phase_change_penalty
  + w_pass * passed_vehicle_count
```

设计要点：

- 延误、排队、等待时间应为负向奖励。
- 通过车辆数、车道疏解量可作为正向奖励。
- 信号切换过快应惩罚，避免模型学到频繁切灯。
- 相位持续时间过短会扰动车流，过长会饿死其他方向车流，两者都应受到约束。
- 奖励量级应稳定，避免某一项过大导致训练不收敛。

最终提交或汇报时，建议包含：

- 作业目标说明：说明智能体要优化单路口信号灯调度。
- 环境与数据说明：列出配置项、观测字段和动作空间。
- 方法说明：描述特征构造、奖励设计、模型结构和算法选择。
- 训练流程说明：解释 workflow、episode、样本收集和模型更新。
- 实验指标说明：列出平均延误、排队长度、等待时间、切换惩罚、reward 和 loss。
- 结果分析：对比不同奖励权重、模型结构或算法的训练效果。
- 问题与改进：讨论频繁切灯、探索不足、样本效率低、模型不稳定等问题。
